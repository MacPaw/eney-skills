The typical directory structure of a newly created extension looks like this:

```bash
my-extension/
├── manifest.json
├── package.json
├── tsconfig.json
├── index.ts
├── components/
│   └── my-widget.tsx
└── tests/
```

## Sources

The entry point is `index.ts` at the root of the extension. It creates the MCP server, registers all widgets, and starts the stdio transport. Each widget is a separate file in `components/` and is imported by `index.ts`.

## Support files

* **manifest.json** is the extension manifest. It describes the extension to Eney: its name, version, author, and how to start the server. See the [field reference](#manifestjson) below.
* **package.json** is a standard npm package file. It must include a `build` script — the CLI calls it when running `eney-skills-cli dev`. See [package.json](#packagejson) below.
* **tsconfig.json** configures TypeScript for ESM output, which is required by the MCP runtime. Most likely, you don't need to edit this file.
* **tests/** is a folder created by the scaffolder which holds few basic tests.

## manifest.json

Describes the extension to Eney and the MCP runtime.

```json
{
  "manifest_version": "0.3",
  "name": "my-extension",
  "version": "1.0.0",
  "description": "What this extension does",
  "author": {
    "name": "Your Name"
  },
  "server": {
    "type": "node",
    "entry_point": "index.js",
    "mcp_config": {
      "command": "node",
      "args": ["${__dirname}/index.js"],
      "env": {}
    }
  },
  "license": "MIT"
}
```

| Field | Description |
|---|---|
| `name` | Unique extension identifier (kebab-case) |
| `version` | Semantic version string |
| `description` | Short description shown in Eney |
| `author.name` | Author name |
| `server.entry_point` | Entry point file (usually `"index.js"`) |
| `server.mcp_config.command` | Command used to start the server |
| `server.mcp_config.args` | Arguments passed to the command. `${__dirname}` is a placeholder for the extension directory — Eney substitutes the real path at runtime |
| `server.mcp_config.env` | Environment variables to inject into the server process |

## package.json

Standard npm package file. The required `build` script compiles TypeScript to JavaScript.

```json
{
  "name": "my-extension",
  "version": "1.0.0",
  "type": "module",
  "scripts": {
    "build": "tsc"
  },
  "dependencies": {
    "@eney/api": "latest",
    "@modelcontextprotocol/sdk": "latest"
  },
  "devDependencies": {
    "typescript": "^5"
  }
}
```

<Callout type="warn">
  The script must be named exactly `"build"`. The CLI calls `npm run build` when running `eney-skills-cli dev`.
</Callout>

## tsconfig.json

Configured for ESM output, which is required by the MCP runtime.

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "outDir": ".",
    "strict": true,
    "jsx": "react-jsx",
    "jsxImportSource": "@eney/api"
  },
  "include": ["**/*.ts", "**/*.tsx"],
  "exclude": ["node_modules"]
}
```

<Callout type="info">
  Because the module system is ESM, all local imports must use `.js` extensions even when the source file is `.ts`. For example: `import { runScript } from "./helpers/run-script.js"`.
</Callout>

## index.ts

The server entry point. Creates the MCP server, registers widgets, and starts the stdio transport.

```typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { setupUIXForMCP } from "@eney/api";
import MyWidget from "./components/my-widget.js";

const server = new McpServer(
  { name: "my-extension", version: "1.0.0" },
  { capabilities: { logging: {}, resources: {} } },
);

const uixServer = setupUIXForMCP(server);

uixServer.registerWidget(MyWidget);

async function main() {
  const transport = new StdioServerTransport();
  await server.connect(transport);
}

main().catch(console.error);
```

To register multiple widgets, import each one and call `uixServer.registerWidget()` for each.

## components/

Contains React widget components. Each file exports a widget created with `defineWidget`. Use `.tsx` for components with JSX. See the [Widgets](/docs/widgets) reference for all available UI primitives.
