### Node.js MCP Extension Structure

Eney extension is a standalone Node.js server that lives in `mcps/<extension-name>/`. Here's the standard layout:

```
mcps/<extension-name>/
├── manifest.json          # Extension metadata & server config
├── package.json           # Node.js dependencies & scripts
├── tsconfig.json          # TypeScript configuration
├── index.ts               # MCP server entry point
└── components/
    ├── my-widget.tsx      # Widget component
    └── helpers.ts         # Optional business logic
```

## manifest.json

Describes the extension for the Eney app and defines how to launch the server.
You can create this file with `mcpb init` command.
Read more about manifest fields in the [official MCPB docs](https://github.com/modelcontextprotocol/mcpb/blob/main/MANIFEST.md).


```json
{
  "manifest_version": "0.3",
  "name": "my-extension-mcp",
  "version": "1.0.0",
  "description": "What the extension does.",
  "author": {
    "name": "Author Name"
  },
  "server": {
    "type": "node",
    "entry_point": "index.js",
    "mcp_config": {
      "command": "node",
      "args": ["${__dirname}/index.js"],
      "env": {}
    }
  },
  "license": "MIT"
}
```

## index.ts

The entry point creates an MCP server, wraps it with the Eney UIX layer, and registers all widgets.

```ts
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { setupUIXForMCP } from "@eney/api";
import MyWidget from "./components/my-widget.js";

const server = new McpServer(
  { name: "my-extension-mcp", version: "1.0.0" },
  { capabilities: { logging: {}, resources: {} } },
);

const uixServer = setupUIXForMCP(server);
uixServer.registerWidget(MyWidget);

async function main() {
  const transport = new StdioServerTransport();
  await server.connect(transport);
  console.error("My Extension running on stdio");
}

main().catch(console.error);
```

Key details:

- `setupUIXForMCP` wraps the MCP server to support widget rendering
- Each widget is registered individually with `uixServer.registerWidget()`
- The server communicates over stdio — `console.error` for logging, `console.log` is reserved for the MCP protocol

### Registering multiple widgets

Add more widgets by importing and registering each one:

```ts
import WidgetA from "./components/widget-a.js";
import WidgetB from "./components/widget-b.js";

uixServer.registerWidget(WidgetA);
uixServer.registerWidget(WidgetB);
```

## Widget Component

Each widget is a React component wrapped with `defineWidget`. Widgets live in the `components/` directory.

```tsx
import { useState } from "react";
import { z } from "zod";
import {
  Action,
  ActionPanel,
  Form,
  Paper,
  defineWidget,
  useCloseWidget,
} from "@eney/api";

const schema = z.object({
  city: z.string().optional().describe("The city name to look up."),
});

type Props = z.infer<typeof schema>;

function MyWidget(props: Props) {
  const closeWidget = useCloseWidget();
  const [city, setCity] = useState(props.city ?? "");
  const [result, setResult] = useState<string | null>(null);

  async function onSubmit() {
    // Business logic — fetch data, compute results, etc.
    setResult(`Result for ${city}`);
  }

  function onDone() {
    closeWidget(result ?? "No result.");
  }

  const actions = (
    <ActionPanel layout="row">
      <Action.SubmitForm title="Submit" onSubmit={onSubmit} style="secondary" />
      <Action title="Done" onAction={onDone} style="primary" />
    </ActionPanel>
  );

  return (
    <Form actions={actions}>
      <Form.TextField name="city" label="City" value={city} onChange={setCity} />
      {result && <Paper markdown={result} />}
    </Form>
  );
}

const MyWidgetDef = defineWidget({
  name: "my-widget",
  description: "Description used by the LLM to select this tool",
  schema,
  component: MyWidget,
});

export default MyWidgetDef;
```

### defineWidget

The `defineWidget` function wraps a component into a registerable widget:

| Field | Description |
|-------|-------------|
| `name` | Kebab-case identifier — used in deeplinks and tool registration |
| `description` | Natural language description — the LLM reads this to decide when to invoke the tool |
| `schema` | Zod schema defining the input props |
| `component` | The React component function |

### Schema

The schema defines what the LLM can pass to the widget. Use Zod with `.describe()` on every field so the LLM understands what each parameter means:

```ts
const schema = z.object({
  query: z.string().optional().describe("Search query to look up."),
  limit: z.number().optional().describe("Maximum number of results."),
  includeDetails: z.boolean().optional().describe("Whether to include detailed info."),
});
```

### useCloseWidget

Call `closeWidget(message)` to close the widget and return a text summary to the AI conversation:

```ts
const closeWidget = useCloseWidget();

function onDone() {
  closeWidget("Here is what happened...");
}
```
